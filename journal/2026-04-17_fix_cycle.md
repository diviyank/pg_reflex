## What shipped

Four code changes, all regression-tested.

### 1. Target-table refresh emits split statements (was: sibling-CTE)

`src/trigger.rs` — `reflex_build_delta_sql`. Three PG17+ `#[cfg]` branches
that emitted `WITH del AS (DELETE … RETURNING), ins AS (INSERT … RETURNING)
UPDATE …` are gone. Split-statement form (already used on PG15/16) is now
unconditional. Sibling CTEs share a snapshot, so the INSERT couldn't see
the DELETE — any grouped-aggregate IMV with a unique index on its group
key hit `duplicate key value` at flush. Split statements commit DELETE
before INSERT runs, so the unique index is happy.

### 2. `replace_source_with_delta` handles qualified column refs

`src/query_decomposer.rs` — new helper. Two-pass rewrite:
- Pass 1: `<src>.<col>` → `<alias>.<col>`
- Pass 2: standalone `<src>` → `<subquery> AS <alias>` (via existing
  `replace_identifier`, which keeps its original simple-identifier
  contract)

`reflex_flush_deferred` in `src/trigger.rs` now uses the new helper for
all four staging paths (INSERT / DELETE / U_OLD / U_NEW). Without this,
`SELECT stock_transfer_baseline.product_id` in an IMV produced
`(SELECT …) AS __dt.product_id AS "product_id"` — syntax error.

### 3. Warning compares expression, not alias

`src/create_ivm.rs` — the "not in GROUP BY" check used the SELECT alias
to look up GROUP BY membership. So `src.col AS renamed` always failed
the check even when `col` was in GROUP BY. Now uses `col.expr_sql` for
bare-name matching; alias is kept only in the message text.

### 4. Temp-view shim for `__reflex_old_<src>` during flush

Surfaced only after #1 and #2 landed (Bug A was crashing early). The
passthrough DELETE/UPDATE branches in `reflex_build_delta_sql` reference
`__reflex_old_<safe_src>` as a literal relation — that name is a trigger
transition table, visible only inside an IMMEDIATE trigger. In
`reflex_flush_deferred` we're at COMMIT time with no transition table,
so the unconditional DELETE SQL hit `relation does not exist`.

Fix: `reflex_flush_deferred` now stands up a temp view
`__reflex_old_<safe_src> AS SELECT * FROM __reflex_delta_<safe_src>
WHERE __reflex_op IN ('D', 'U_OLD')` before the per-IMV loop and drops
it after. Same schema, right content, zero API surface changes.

## New bugs discovered downstream (out of this cycle)

After landing the four fixes, the end-to-end flush on db_clone reveals
two more pre-existing bugs that were masked by Bug A:

- **`build_min_max_recompute_sql` doesn't carry LEFT JOINs.** When an
  IMV's BOOL_OR / MIN / MAX source arg references a joined alias like
  `caav.product_id` (from `LEFT JOIN current_assortment_activity_reflex
  caav`), the recompute emits `UPDATE intermediate SET col = (SELECT
  BOOL_OR(caav.…) FROM <source_table> WHERE …)` — `caav` is not in that
  FROM and Postgres rejects it. Affects `unsent_sop_forecast_reflex`,
  `sop_forecast_reflex`, `sop_last_forecast_reflex`,
  `demand_planning_characteristics_reflex`,
  `stock_transfer[_baseline]_reflex`, and any other IMV with a
  BOOL_OR/MIN/MAX over a joined column. File: `src/trigger.rs:248`
  (`build_min_max_recompute_sql`).
- **Cascading CTE-IMV identifier truncation.** Benign but noisy:
  `__reflex_old_demand_planning_characteristics_reflex__cte_sales_stats`
  is longer than 63 chars and gets silently truncated. Doesn't
  (apparently) break anything here but risks name collisions in deeper
  graphs.

Neither is in scope of this fix cycle. Journal them as follow-ups.
